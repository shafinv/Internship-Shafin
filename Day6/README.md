# FIFO Transaction Documentation

## Overview

The **transaction** is a SystemVerilog class that represents **one FIFO operation**. It is used by the verification environment to transfer data between the Generator, Driver, Monitor, and Scoreboard.

A transaction is a **high-level representation** of the DUT inputs and outputs. It does **not** generate the clock or reset signals and does **not** directly communicate with the DUT.

---

# Transaction Class

```systemverilog
class transaction;

    // DUT Inputs
    rand bit        wrenb;
    rand bit        rdenb;
    rand bit [7:0]  data_in;

    // DUT Outputs
    bit [7:0] data_out;
    bit full;
    bit empty;

    // Read and Write cannot occur simultaneously
    constraint wr_rd {
        !(wrenb && rdenb);
    }

    function void display(string tag);
        $display("[%s]", tag);
        $display("WR_EN    = %0b", wrenb);
        $display("RD_EN    = %0b", rdenb);
        $display("DATA_IN  = %0h", data_in);
        $display("DATA_OUT = %0h", data_out);
        $display("FULL     = %0b", full);
        $display("EMPTY    = %0b", empty);
    endfunction

endclass
```

---

# Transaction Members

| Variable   | Type            | Description                                       |
| ---------- | --------------- | ------------------------------------------------- |
| `wrenb`    | `rand bit`      | Write Enable signal. Randomized by the generator. |
| `rdenb`    | `rand bit`      | Read Enable signal. Randomized by the generator.  |
| `data_in`  | `rand bit[7:0]` | Data written into the FIFO.                       |
| `data_out` | `bit[7:0]`      | Data read from the FIFO. Filled by the monitor.   |
| `full`     | `bit`           | Indicates FIFO Full status. Captured from DUT.    |
| `empty`    | `bit`           | Indicates FIFO Empty status. Captured from DUT.   |

---

# Why are only some variables randomized?

Only the **input signals** are randomized because they are generated by the testbench.

```text
Generator
      │
      ▼
Transaction
      │
      ▼
Driver
      │
      ▼
FIFO DUT
```

The DUT produces the output signals.

```text
FIFO DUT
      │
      ▼
Monitor
      │
      ▼
Transaction
```

Therefore:

* `wrenb` → Randomized
* `rdenb` → Randomized
* `data_in` → Randomized
* `data_out` → Not randomized
* `full` → Not randomized
* `empty` → Not randomized

---

# Why are Clock and Reset not included?

The transaction represents **only one FIFO operation**.

The clock and reset belong to the **interface/testbench**, not to an individual transaction.

Clock generation:

```systemverilog
always #5 clk = ~clk;
```

Reset sequence:

```systemverilog
rst = 1;
#20;
rst = 0;
```

Since these are global signals, they are **not part of the transaction class**.

---

# Constraint

```systemverilog
constraint wr_rd {
    !(wrenb && rdenb);
}
```

This constraint prevents simultaneous read and write operations.

Allowed combinations:

| `wrenb` | `rdenb` | Operation |
| ------- | ------- | --------- |
| 0       | 0       | Idle      |
| 1       | 0       | Write     |
| 0       | 1       | Read      |
| 1       | 1       | Illegal   |

# Transaction Flow

```
Generator
     │
     ▼
Creates Transaction
     │
     ▼
Driver
     │
     ▼
FIFO DUT
     │
     ▼
Monitor
     │
     ▼
Updates Transaction
     │
     ▼
Scoreboard
```

# Example Transaction

### Write Transaction

```
WR_EN   = 1
RD_EN   = 0
DATA_IN = AA
```

Meaning:

```
Write 0xAA into the FIFO.
```

---

### Read Transaction

```
WR_EN   = 0
RD_EN   = 1
```

Meaning:

```
Read one data item from the FIFO.
```

The monitor later updates:

```
DATA_OUT = AA
FULL     = 0
EMPTY    = 0
```

---

# Important Notes

* A transaction is a **class**, not a module.
* One object represents **one FIFO operation**.
* The generator randomizes only the input fields.
* The monitor fills the output fields after observing the DUT.
* The transaction does not contain timing information such as `clk` or `rst`.
* The same transaction object type is shared by the Generator, Driver, Monitor, and Scoreboard.

---

# Summary

The transaction class is the central data structure used in the FIFO verification environment. It encapsulates all information required for a single FIFO operation and provides a clean interface for communication between verification components. This layered approach improves modularity, reusability, and scalability, and forms the basis of modern verification methodologies such as UVM.
